Switch & LED Blinking with STM32G070RB (Low Power)

Problem statement

A tactile switch and LED connected to a NUCLEO-G070RB cycle through blink rates on each press:
1st press → 0.5 Hz, 2nd press → 1 Hz, 3rd press → 2 Hz, 4th press → off, 5th press → repeats from 0.5 Hz. The design must use the microcontroller's low-power modes to minimize power consumption.

Design approach


Register-level (CMSIS) implementation, no HAL — direct manipulation of GPIO, EXTI, RCC, TIM3, and NVIC registers.
Button input is handled entirely by interrupt (EXTI line 13, falling edge), not polled — the CPU does no work waiting for a press.
Debouncing is done in software using a free-running SysTick millisecond counter: any EXTI trigger within 50ms of the last accepted press is treated as switch bounce and ignored, without using a blocking delay.
State machine: a 4-state counter (0=off, 1=0.5Hz, 2=1Hz, 3=2Hz) advances on each accepted press and wraps back to 0 after state 3.
Blink timing is generated by TIM3, configured with a 1ms base tick (PSC=15999 at 16MHz) and a per-state ARR value (999/499/249 ms) that toggles the LED on each timer interrupt — no busy-wait delay loops are used for timing.
Low-power mode: the main loop calls __WFI() (Wait For Interrupt) instead of spinning. The CPU enters Sleep mode between every button press and timer tick, and wakes only when EXTI13 or TIM3's update interrupt fires.


Why Sleep mode (not Stop or Standby)

Stop mode gates peripheral clocks, which would stop TIM3 from counting — breaking the blink mechanism entirely unless the timing were migrated to a Low-Power Timer (LPTIM) capable of running during Stop. Standby mode is more aggressive still and would not preserve current_state or peripheral configuration across wake-ups without additional backup-register handling. Sleep mode was chosen because it keeps TIM3 and EXTI fully functional while still stopping the CPU core clock between events, which is the correct trade-off given the assignment's scope and timeline.

Assumptions


Clock source: the microcontroller runs on its default 16MHz internal RC oscillator (no external crystal or PLL configuration), since the assignment does not specify a required clock source. All PSC/ARR timer values are calculated against this 16MHz base.
Debounce window: 50ms was chosen as a typical, safe debounce interval for a tactile switch; this can be tuned after testing on real hardware.
Blink definition: "frequency X Hz" is interpreted as a full on-off cycle occurring X times per second, meaning the LED toggle itself happens at 2X — e.g. 0.5Hz blinking toggles the LED every 1000ms (1000ms on, 1000ms off).
State 0 (off): in addition to stopping TIM3, the LED is explicitly forced off via a direct GPIO write, rather than just halting the timer mid-cycle (which could leave the LED stuck on if the timer happened to stop while the pin was high).
Low-power scope: only Sleep mode is implemented, since the chosen architecture (TIM3 for blink timing) is incompatible with Stop/Standby modes without further redesign (see above). This is a deliberate, documented trade-off rather than an oversight.
Hardware validation pending: this code has been written and desk-checked (logic traced by hand for all state transitions, debounce timing, and register bit math) but not yet flashed to physical hardware, since the NUCLEO-G070RB board was not available at the time of writing. Hardware validation will be completed before the in-person demo.
